`<nav>`는 “메뉴 바”를 예쁘게 만드는 태그가 아니라 **페이지 안에서 ‘어디로 갈지’를 정의하는 시맨틱 영역**입니다.
바로 말해서: **내비게이션(탐색)을 담당하는 영역**을 브라우저와 보조기술(스크린리더 등)에게 알려주는 역할이에요. 🧭

---

## 1. `<nav>` 한 줄 정의 ✨

> **“사용자가 다른 페이지나 같은 페이지의 중요한 섹션으로 이동할 수 있게 해주는 링크들의 모음 영역”** 을 나타내는 시맨틱 태그.

* “링크 몇 개” → `<a>`
* “링크 목록” → `<ul>`, `<li>`, `<a>`
* “이 영역이 내비게이션이다” → `<nav>`

```html
<nav>
  <ul>
    <li><a href="/">홈</a></li>
    <li><a href="/about">소개</a></li>
    <li><a href="/contact">문의</a></li>
  </ul>
</nav>
```

겉으로는 `<div>`와 똑같이 보일 수 있지만, **브라우저·검색엔진·스크린리더에게는 완전히 다른 정보**를 전달합니다. 🧠

---

## 2. 왜 굳이 `<div>` 말고 `<nav>`를 써야 할까? 🧐

### 2.1 시맨틱(의미) 정보 전달

* `<div>`: 의미 없음, 그냥 박스 🧱
* `<nav>`: “여기는 내비게이션 영역입니다” 라고 **역할을 선언**.

이 덕분에:

* **검색엔진(SEO)**: 페이지 구조 파악에 도움
* **스크린리더 사용자**: “내비게이션 영역으로 바로 이동” 같은 기능 사용 가능
* **유지보수하는 개발자**: 코드를 읽을 때 역할이 한눈에 보임

### 2.2 접근성(Accessibility) 향상 ♿

스크린리더는 페이지를 읽을 때 **랜드마크(landmark) 영역**으로 이동할 수 있어요.

* `<nav>`는 **기본적으로 `role="navigation"` 랜드마크** 역할을 가집니다.
* 키보드만 사용하는 사용자는 “다음 내비게이션으로 이동” 같은 명령으로 빠르게 점프 가능.

```html
<header>
  <a href="#main" class="skip-link">본문 바로가기</a>
  <nav aria-label="주 내비게이션">
    ...
  </nav>
</header>

<main id="main">
  ...
</main>
```

여기서 `aria-label="주 내비게이션"`은 스크린리더에 “Main navigation” 같은 이름으로 읽어주도록 합니다.

### 2.3 SEO 관점 🔍

SEO 자체가 `<nav>` 하나로 dramatically 좋아진다! 이런 건 아니지만:

* 검색엔진이 **페이지의 구조와 주요 링크들을 더 잘 이해**함
* 메뉴 / 푸터 내비게이션 등의 패턴을 더 정확히 인식

→ 결과적으로 **크롤링·인덱싱에 유리한 구조**를 만들 수 있습니다.

## 3. `<nav>`는 어디에 써야 할까? (올바른 사용 범위) ✅

HTML Living Standard에서는 `<nav>`를 다음과 같이 권장합니다:

> “문서나 사이트의 주요 내비게이션 링크 모음을 감쌀 때 사용하라.”

즉:

### ✅ 대표적인 사용 예

1. **사이트 전체를 이동하는 메인 메뉴**

```html
<header>
  <h1>My Blog</h1>
  <nav aria-label="주 메뉴">
    <ul>
      <li><a href="/">홈</a></li>
      <li><a href="/posts">글 목록</a></li>
      <li><a href="/tags">태그</a></li>
      <li><a href="/about">소개</a></li>
    </ul>
  </nav>
</header>
```

2. **섹션 내 서브 메뉴 (탭, 카테고리 등)**

```html
<section>
  <h2>내 계정</h2>
  <nav aria-label="내 계정 메뉴">
    <ul>
      <li><a href="/account/profile">프로필</a></li>
      <li><a href="/account/security">보안 설정</a></li>
      <li><a href="/account/billing">결제 정보</a></li>
    </ul>
  </nav>

  <article>
    <!-- 현재 선택된 메뉴 내용 -->
  </article>
</section>
```

3. **페이지 내 목차 (Table of Contents)**

```html
<aside>
  <nav aria-label="문서 목차">
    <ol>
      <li><a href="#intro">1. 소개</a></li>
      <li><a href="#setup">2. 설치</a></li>
      <li><a href="#usage">3. 사용법</a></li>
    </ol>
  </nav>
</aside>
```

### ❌ 사용하지 않는 게 좋은 경우

* 그냥 **본문 중에 널려 있는 링크** (문장 중의 링크들)
* “다음 글 보기” 정도의 **단일 링크** 하나
* 광고 배너에 달린 링크들

이런 것들은 **‘내비게이션’이라 부르기 애매**한 경우라 `<nav>`보다 그냥 `<a>` 또는 `<aside>` + `<a>` 등에 가까워요.

---

## 4. `<nav>`와 다른 태그들의 관계 정리 🧩

### 4.1 `<nav>` vs `<header>`

* `<header>`: “이 섹션의 머리말, 소개, 로고, 검색폼 등”
* `<nav>`: “링크 모음(내비게이션)”

보통은 **header 안에 nav**를 두는 패턴이 많습니다:

```html
<header>
  <h1>My Site</h1>
  <nav>
    ...
  </nav>
</header>
```

하지만 필요하다면:

```html
<header>...</header>
<nav>...</nav>
<main>...</main>
```

처럼 **형제 요소로 분리**해도 전혀 문제 없습니다.

---

### 4.2 `<nav>` vs `<footer>`

* `<footer>`에도 자주 “푸터 메뉴”가 들어갑니다.
* 그 메뉴 영역에 `<nav>`를 감싸는 것은 **좋은 패턴**입니다.

```html
<footer>
  <nav aria-label="푸터 메뉴">
    <ul>
      <li><a href="/terms">이용약관</a></li>
      <li><a href="/privacy">개인정보 처리방침</a></li>
      <li><a href="/contact">문의하기</a></li>
    </ul>
  </nav>
  <p>© 2026 My Company</p>
</footer>
```

---

### 4.3 `<nav>` vs `<aside>`

* `<aside>`: “본문의 흐름에서 약간 벗어난 보조 콘텐츠”
* `<nav>`: “내비게이션 역할”

사이드바 내비게이션은 두 개를 함께 쓰기도 합니다:

```html
<aside>
  <nav aria-label="사이드바 내비게이션">
    ...
  </nav>
</aside>
```

---

### 4.4 `<nav>` vs `<ul>/<ol>`

* `<nav>`는 **컨테이너** (영역)
* `<ul>` / `<ol>`은 **리스트 구조**

관계는 보통 이렇게:

```html
<nav>
  <ul>
    <li><a href="...">...</a></li>
    <li><a href="...">...</a></li>
  </ul>
</nav>
```

**중요 포인트**:
**`<nav>`가 리스트를 강제하진 않습니다.**
단지 내비게이션 역할만 정의할 뿐.
그러나 **접근성·구조상 대부분의 내비게이션은 리스트로 표현**하는 것이 좋습니다.

---

## 5. `<nav>`는 여러 개 써도 될까? (여러 내비게이션) 🧭🧭

**네. 한 문서에 `<nav>`가 여러 개 있어도 완전 괜찮습니다.**

예시:

```html
<header>
  <nav aria-label="주 메뉴">...</nav>
</header>

<aside>
  <nav aria-label="카테고리 메뉴">...</nav>
</aside>

<footer>
  <nav aria-label="푸터 메뉴">...</nav>
</footer>
```

이렇게 하면 스크린리더가:

* “주 메뉴 네비게이션”
* “카테고리 메뉴 네비게이션”
* “푸터 메뉴 네비게이션”

처럼 구분해서 읽어줄 수 있습니다.

따라서 **`aria-label`로 각각의 역할을 명확하게 이름 붙여주는 게 베스트 프랙티스**입니다. 💡

---

## 6. ARIA와 `<nav>`: role="navigation"은 언제? 🎛️

`<nav>`는 기본적으로 **`role="navigation"`이 내장**되어 있어서 보통은 굳이 다시 쓸 필요 없습니다.

```html
<nav> <!-- role="navigation" 내장 -->
  ...
</nav>
```

하지만:

* `<div>`인데 내비게이션 역할을 하고 있다면?

```html
<div role="navigation" aria-label="주 메뉴">
  ...
</div>
```

이렇게 `role="navigation"`을 명시해야 합니다.
그래서 **내비게이션이라면 `<div>` + `role="navigation"`보다 `<nav>`를 쓰는 게 훨씬 깔끔**해요.

---

## 7. Skip Link + `<nav>` 패턴 (실전 접근성) ⌨️

키보드만 사용하는 사용자를 위해서, **맨 위에 “본문으로 바로가기” 링크**를 두는 패턴이 자주 쓰입니다.

```html
<a href="#main" class="skip-link">본문 바로가기</a>

<header>
  <nav aria-label="주 메뉴">
    ...
  </nav>
</header>

<main id="main">
  ...
</main>
```

CSS 예시:

```css
.skip-link {
  position: absolute;
  left: -9999px;
}

.skip-link:focus {
  left: 0;
  top: 0;
  padding: 1rem;
  background: black;
  color: white;
  z-index: 1000;
}
```

* 평소에는 안 보임
* `Tab` 키로 포커스를 받으면 화면 상단에 표시 → 바로 `<main>`으로 점프
  → `<nav>`에서 한참 Tab으로 메뉴를 건너뛰지 않아도 되니까 **접근성이 크게 향상**됩니다. 🙌

---

## 8. 반응형 내비게이션 + `<nav>` (햄버거 메뉴) 🍔

실제 프로젝트에서는 `<nav>`를 이렇게 React/Tailwind 등과 결합해서 많이 씁니다.

### HTML 예시

```html
<header>
  <div class="logo">My Site</div>

  <button aria-expanded="false" aria-controls="main-menu" class="menu-toggle">
    ☰
    <span class="sr-only">메뉴 열기</span>
  </button>

  <nav id="main-menu" aria-label="주 메뉴">
    <ul>
      <li><a href="/">홈</a></li>
      <li><a href="/products">상품</a></li>
      <li><a href="/cart">장바구니</a></li>
      <li><a href="/mypage">마이페이지</a></li>
    </ul>
  </nav>
</header>
```

* `<nav>`는 여전히 “내비게이션” 역할만 담당
* 햄버거 버튼은 `aria-expanded` / `aria-controls`로 열림/닫힘 상태를 전달
* CSS/JS/React로 show/hide를 제어

---

## 9. React에서 `<nav>` 사용 예 (React Router 포함) ⚛️

선생님께서 React & React Router 강의도 하시니까, 바로 연결할 수 있는 예제를 보겠습니다. 😉

```jsx
import { Link } from 'react-router-dom'

export default function Navbar() {
  return (
    <header className="flex items-center justify-between px-4 py-2 shadow">
      <h1 className="text-xl font-bold">
        <Link to="/">My App</Link>
      </h1>

      <nav aria-label="메인 내비게이션">
        <ul className="flex gap-4">
          <li>
            <Link to="/" className="hover:underline">
              홈
            </Link>
          </li>
          <li>
            <Link to="/about" className="hover:underline">
              소개
            </Link>
          </li>
          <li>
            <Link to="/posts" className="hover:underline">
              게시글
            </Link>
          </li>
        </ul>
      </nav>
    </header>
  )
}
```

여기서도 `nav`의 역할은 **“이 영역은 내비게이션이다”**를 시맨틱하게 표현하는 것뿐이고, 라우팅은 React Router가 담당합니다.

## 10. `<nav>` 사용 시 자주 하는 실수 ⚠️

1. **페이지에 있는 모든 링크들을 `<nav>`로 감싸는 경우**

   * 본문 중의 링크까지 싹 다 내비게이션으로 해버리면:
     스크린리더에게 “이 문서 전체가 내비게이션”처럼 들릴 수 있음 → 구조가 망가짐

2. **내비게이션인데 `<div>`만 사용하는 경우**

   ```html
   <div class="nav">
     <a href="/">Home</a>
     <a href="/about">About</a>
   </div>
   ```

   → 눈에는 똑같이 보이지만, 브라우저/보조기술에는 **‘그냥 박스’**로만 보임.

3. **여러 `<nav>`를 쓰면서 label을 안 붙이는 경우**

   * `<nav>`가 여러 개 있을 때 이름이 없으면, 스크린리더는 “navigation, navigation, navigation…”
   * `aria-label`로 구분: `"주 메뉴"`, `"사이드 메뉴"`, `"푸터 메뉴"` 등.

4. **단일 링크에 `<nav>`를 쓰는 경우**

   * “이전 글” 링크 하나, “맨 위로” 링크 하나 정도는 `<nav>`로 감싸지 않는 것이 일반적.

---

## 11. 짧은 요약 💡

* `<nav>`는 **“내비게이션 링크들의 시맨틱 컨테이너”**입니다.
* 브라우저·검색엔진·보조기술에 **‘여기가 메뉴/탐색 영역이다’**라는 정보를 전달합니다.
* 한 페이지에 여러 개 써도 괜찮지만, 가급적 **주요 내비게이션 링크 모음에만 사용**합니다.
* React/SPA 환경에서도 여전히 중요하며, 특히 **접근성과 구조적인 코드**를 위해 필수적인 태그입니다.
